Effで作った、プログラム固有のデータ型（評価器とか）は抽象化しないのが一番楽そう

Posted on 2017/07/14
Tags: [haskell]
  • Tweet

　最近は僕のLisp処理系のリポジトリでextensible-effectsの実験をしていて、 今のところ「Haskellはeffectsよりもmonad-transformerだなあ」という感じ。

　そこで今一番ムズいところというと…… 僕は評価器を作るにあったって、MaruEvaluatorという評価器を表す型を、Rank2Typesを使って作った。

type ExceptionCause = Text      -- 処理が失敗した理由
type Fail' = Exc ExceptionCause -- 処理が失敗するかもしれないという作用（effect）
-- MaruEnvは、定義された変数や関数の実データを持つ辞書
type MaruEvaluator a = forall r. ( Member Fail' r
                                 , Member (State MaruEnv) r
                                 , SetMember Lift (Lift IO) r
                                 ) => Eff r a

-- SExprはS式
eval :: SExpr -> MaruEvaluator SExpr

　ここまでは問題ないんだけど、eval内ではMaruEvaluatorの性質を弱めた作用を持つ関数…… 例えばこんなこと

-- TextはMaruEnv辞書の鍵であって、シンボルを表す。
-- lookupSymbolは鍵を受け取り、状態の中からシンボルの実値（SomeMaruPrimitive）
--                                                   を引き出す（State MaruEnv）。
-- ただし受け取った鍵を、状態が持たないかもしれないので失敗する可能性がある（Fail'）。
lookupSymbol :: (Member (State MaruEnv) r, Member Fail' r) => Text -> Eff r SomeMaruPrimitive

-- `Cons :: SExpr -> SExpr -> SExpr`,
-- `Symbol :: Text -> SExpr`
-- はSExprの値構築子
eval (Cons (Symbol x) y) = do
    val <- lookupSymbol x
